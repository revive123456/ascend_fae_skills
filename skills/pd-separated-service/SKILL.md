# PD 分离服务拉起

## 核心原则

- **以文档为准**：启动顺序、配置模板、LB 方案均从镜像 `/vllm-workspace/` 或 vllm-ascend 仓库提取，不做假设。
- **分阶段 + 检查点**：4 个阶段（信息收集 → 脚本生成 → 服务拉起 → 验证诊断），每阶段结束需用户确认后才进入下一阶段。
- **尽力自动修复**：环境类、配置类、代码 bug 均在环境上尝试修复。同节点每类报错最多重试 2 次，超限或不可修则输出诊断结论并结束。代码 bug 同样属于可修复问题——定位源码后尝试打补丁。
- **特性开关不擅自改**：CP 使能等功能开关保持原样，改动须征得用户同意。环境配置按文档要求核对，不以"默认应该开"做假设。
- **诊断透明**：每一步告知用户查了什么 → 定位到哪 → 结论是什么。

## 4 阶段流程

### 阶段 1：信息收集

逐个收集必须参数（每个参数单独提问，缺一不可）：

| 参数 | 必须？ | 说明 |
|------|--------|------|
| 拓扑类型 | ✅ | 单机多卡 / 多机分离 / 用户自述 |
| P 节点 IP | ✅ | Prefill 节点 IP 列表 |
| D 节点 IP | ✅ | Decode 节点 IP 列表 |
| 权重路径 | ✅ | 节点共享路径或各自路径 |
| Docker 镜像 | ✅ | PD 分离所用镜像地址 |

高级参数（max-model-len、tensor-parallel-size、pipeline-parallel-size、dtype 等）不主动追问，用户未提供则使用文档默认值。

**预检**：收集到 IP 后立即测 SSH 连通性；收集到权重路径后检查是否存在。不通或不存在则暂停等待用户。

**阶段 1 输出**：参数清单 → 用户确认后进入阶段 2。

### 阶段 2：脚本生成

1. **读文档**：
   - 优先：`docker run --rm <image> cat /vllm-workspace/` 读取镜像内文档
   - 降级：查 vllm-ascend 仓库 README / docs
   - 兜底：用模型自身领域知识生成，标注"基于已知知识，请对照实际文档确认"
2. **生成脚本**：P 节点启动脚本、D 节点启动脚本、LB 配置/脚本（优先复用文档中的 load_balancer）。脚本顶部注释区分"用户指定参数"与"文档默认值"。
3. **展示开关清单**：列出脚本中所有开关项（CP、RoCE、prefix caching、chunked prefill 等），标注状态（开/关）和来源（用户指定/文档默认）。用户审查通过后进入阶段 3。

**阶段 2 输出**：脚本 + 开关清单 → 用户确认。

### 阶段 3：服务拉起

按文档确定的顺序 SSH 远程执行启动。

**报错分类处理**：

| 类型 | 示例 | 处理 |
|------|------|------|
| 环境 | 端口占用、Docker 未启 | 自动修复 → 重试 |
| 配置 | 镜像未 pull、路径不存在 | 查源码定位 → 可修则修 → 重试；不可修则输出建议 → 结束 |
| 代码 | vLLM 内部报错、算子异常 | 查 vllm-ascend 源码定位 → 打补丁修复 → 重试 |

同节点最多重试 2 次，超限或不可自动修复 → 输出诊断结论（日志 + 源码位置 + 已尝试修复 + 建议）→ 结束。

**阶段 3 输出**：
- 成功：所有节点进程状态 / 容器 ID
- 失败：失败节点 + 报错日志 + 源码定位 + 已尝试修复 + 下一步建议

### 阶段 4：验证与诊断

**逐层验证**（从底层到顶层，任一层失败则停止进入诊断）：

```
D 节点健康检查 → P 节点健康检查 → LB 入口 → curl /v1/chat/completions
```

验证通过 → 输出可复用 curl 示例和实际响应 → 结束。

验证失败 → 收日志（容器日志/dmesg）→ 查 vllm-ascend 源码定位 → 分类：
- 可修（环境/配置/代码 bug）→ 修复 → 回到阶段 3 重拉服务
- 不可修（硬件等）→ 输出诊断链路 → 结束

**诊断汇报格式**：
> "验证 D 节点 X.X.X.X:8000 时请求超时。查容器日志发现 XXXError at vllm_ascend/xxx.py:123。尝试 YYY 修复...[结果]。结论：[根因 + 建议]。"

**终止条件**：完整 curl 返回正常响应 → 全部结束，输出服务入口地址。不可修复 → 输出诊断链路 → 结束。

## 基线测试场景

| # | 输入 | 期望行为 |
|---|------|----------|
| 1 | "拉起 PD 分离服务" | 反问拓扑类型 |
| 2 | 提供拓扑但缺 P 节点 IP | 反问缺失参数 |
| 3 | 单机多卡，参数齐全 | 生成 P/D 脚本 → 展示开关 → 等用户确认 |
| 4 | 多机分离，参数齐全 | 生成 P/D/LB 脚本 → 展示开关 → 等用户确认 |
| 5 | 阶段 3 Docker Daemon 未启动 | 自动启动 Docker → 重试 |
| 6 | 阶段 3 权重路径不存在 | 提示用户先下载或修正路径 |
| 7 | 阶段 3 端口冲突 | 释放或换端口 → 重试 |
| 8 | 阶段 3 重试 2 次仍失败 | 输出完整诊断链路 → 结束 |
| 9 | 阶段 4 curl /v1/chat/completions 正常 | 输出服务入口 + curl 响应样例 |
| 10 | 阶段 4 验证失败（代码 bug） | 查日志 → 查源码 → 补丁 → 回阶段 3 |
| 11 | 文档获取全部失败 | 用模型知识生成脚本，标注"未验证" |
| 12 | 用户主动提供高级参数 | 注入脚本，标注"用户指定" |
